# 상품 상세 조회 캐시 전략

## 📋 배경

상품 상세 조회는 사용자가 반복적으로 많이 호출하는 API 중 하나였고, 상품 정보, 이미지, 좋아요 여부처럼 서로 성격이 다른 데이터를 함께 조합해 응답하고 있었다.
특히 상품 자체 정보와 이미지는 공통 데이터인 반면, 좋아요 여부는 사용자마다 달라지는 데이터라서 같은 방식으로 다루기 어려웠다.


---
## ⚠️ 문제 상황

초기 구조에서는 상품 상세 조회 시 다음 데이터가 매번 DB에서 함께 조회되고 있었다.

- 상품 및 브랜드 정보
- 상품 이미지 목록
- 사용자별 좋아요 여부

이 구조는 기능적으로는 단순했지만, 공통 데이터와 사용자별 데이터가 한 요청 안에서 함께 섞여 있었다.
그 결과 동일한 상품이 반복 조회되더라도 상품·브랜드·이미지 같은 공통 데이터까지 매번 다시 조회하게 됐다.

실제 Hibernate SQL 로그를 보면 캐시 미적용 시 요청마다 다음 쿼리가 반복적으로 실행됐다.

- 상품 + 브랜드 조회
- 상품 이미지 조회
- 좋아요 존재 여부 조회

문제의 핵심은 **변하지 않는 공통 데이터와 사용자별 실시간 데이터를 같은 비용으로 반복 조회하고 있었다**는 점이었다.


---
## 💡 해결 방향

핵심 판단은 다음과 같았다.

1. **공통 데이터와 사용자별 데이터는 캐시 전략을 다르게 가져가야 한다.**
2. **이번 최적화의 목적은 DB 조회 감소 효과를 순수하게 검증하는 것이다.**
3. **효과를 명확히 검증하기 쉽고 책임 분리가 분명한 상세 조회부터 캐시하는 편이 적절하다.**

실무에서는 목록 조회에 캐시를 먼저 적용하는 경우가 많지만, 이번에는 공통 데이터와 사용자별 데이터를 분리하는 전략을 더 분명하게 보여줄 수 있는 상세 조회를 우선 대상으로 삼았다.

이를 바탕으로 상품 상세 조회 응답을 두 종류로 나눴다.

- 상품, 브랜드, 이미지 → 공통 데이터
- 좋아요 여부 → 사용자별 데이터

이 구조에서는 캐시 가능한 데이터와 실시간으로 남겨둘 데이터를 명확히 분리했다.


---
## 🔧 구현

### 1. 공통 데이터와 사용자별 데이터 분리

상품 상세 조회에서 다음 데이터는 캐시 대상으로 분리했다.

- 상품 정보
- 브랜드 정보
- 이미지 목록

반면 좋아요 여부는 사용자마다 다르고 요청 시점마다 달라질 수 있으므로, 캐시하지 않고 매번 조회하도록 남겼다.

이렇게 하면 공통 데이터는 반복 조회 시 캐시에서 재사용하고, 사용자별 상태만 실시간으로 확인할 수 있다.

### 2. Caffeine 선택

이번 최적화의 목적은 **DB 조회 감소가 응답 시간에 어떤 효과를 주는지 검증하는 것**이었다.
이 상황에서 Redis를 사용하면 네트워크 I/O 비용과 역직렬화 비용이 측정치에 섞이기 때문에, 캐시 자체의 순수 효과를 보기 어렵다.

따라서 단일 서버 환경에서는 네트워크 비용이 없는 로컬 인메모리 캐시인 Caffeine을 선택했다.

이 판단의 기준은 명확했다.

- 현재는 단일 서버 환경이다.
- 캐시 접근 비용을 최소화하고 싶었다.
- 먼저 로컬 캐시로 병목을 제거한 뒤, 이후 수평 확장 시 Redis 전환을 고려할 수 있다.

Caffeine은 **현재 운영 환경과 검증 목적에 맞춘 기술 선택**이었다.

### 3. 확장 가능성 고려

캐시 계층은 이후 Redis로 전환할 수 있도록 분리 가능한 구조를 전제로 설계했다.
지금은 Caffeine으로 상세 조회 캐시 효과를 검증하고, 이후 서버 수평 확장이나 캐시 일관성 요구가 커지면 Redis 기반 분산 캐시로 옮길 수 있는 방향을 열어두었다.


---
## 📊 측정 및 검증

동일 환경에서 50회 반복 측정한 결과는 다음과 같았다.

- 캐시 미적용 총 시간: 552ms
- 캐시 적용 총 시간: 160ms

즉, 평균 응답 시간 기준으로 약 **71% 개선**을 확인했다.

### SQL 로그 검증

캐시 미적용 시에는 요청마다 다음 세 쿼리가 반복됐다.

- 상품 + 브랜드 조회
- 상품 이미지 조회
- 좋아요 여부 조회

캐시 적용 후에는 공통 데이터가 캐시에서 조회되고, 사용자별 데이터인 좋아요 여부 조회만 남았다.
캐시 효과는 응답 시간뿐 아니라 **실제 DB 접근 감소**로도 확인할 수 있었다.

#### 캐시 미적용 — 요청마다 3개 쿼리 반복 실행

```sql
-- 1. 상품 + 브랜드 조회 (fetch join)
select p1_0.id, p1_0.name, p1_0.content, p1_0.cost_price, p1_0.discount_rate,
       p1_0.thumbnail_img, p1_0.created_at, p1_0.updated_at,
       b1_0.id, b1_0.name, b1_0.image_url
from product p1_0
join brand b1_0 on b1_0.id = p1_0.brand_id
where p1_0.id = ?

-- 2. 상품 이미지 조회
select pi1_0.id, pi1_0.product_id, pi1_0.url, pi1_0.image_type, pi1_0.sequence
from product_image pi1_0
where pi1_0.product_id = ?
order by pi1_0.sequence asc nulls last

-- 3. 좋아요 여부 조회
select count(l1_0.id) > 0
from likes l1_0
where l1_0.member_id = ?
  and l1_0.target_id = ?
  and l1_0.like_type = 'PRODUCT'
```

#### 캐시 적용 — 공통 데이터 캐시 HIT 시 쿼리 1개만 실행

```sql
-- 좋아요 여부만 실시간 조회
select count(l1_0.id) > 0
from likes l1_0
where l1_0.member_id = ?
  and l1_0.target_id = ?
  and l1_0.like_type = 'PRODUCT'
```

상품·브랜드·이미지 관련 쿼리는 Caffeine 캐시에서 반환되어 DB에 도달하지 않았다.

## ✅ 결과

이 구조를 적용한 뒤 상품 상세 조회는 공통 데이터와 사용자별 데이터의 성격을 분리한 캐시 구조로 정리됐다.

구체적으로는 다음 효과를 얻었다.

- 반복 조회 시 상품·브랜드·이미지 재조회 제거
- 좋아요 여부는 실시간 데이터로 유지
- 상세 조회 응답 시간 약 71% 개선
- SQL 로그로 실제 DB 접근 감소 확인

이 작업을 통해 **효과를 검증하기 쉽고 책임 분리가 명확한 상세 조회부터 단계적으로 적용**하는 방향도 정리할 수 있었다.

## 📝 정리

상품 상세 조회는 공통 데이터와 사용자별 데이터를 분리해 캐시하고, 단일 서버 환경에 맞는 Caffeine을 적용해 DB 조회 비용과 응답 시간을 함께 줄였다. 이후 수평 확장이나 캐시 일관성 요구가 커질 경우 Redis 기반 분산 캐시로 확장할 수 있는 방향도 함께 열어두었다.
