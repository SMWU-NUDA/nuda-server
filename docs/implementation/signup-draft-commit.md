# 회원가입 Draft-Commit 구조

## 📋 배경

기존 회원가입은 단계별 입력에 따라 실제 엔티티를 생성하거나 수정하는 방식으로 구현되어 있었다.
하지만 실제 UI는 사용자가 이전 단계로 돌아가 내용을 수정할 수 있는 편집형 플로우였고, 백엔드 모델은 이를 전제로 설계되어 있지 않았다.


---
## ⚠️ 문제 상황

기존 구조에서는 회원가입 단계를 다음과 같이 처리했다.

- 기본 정보 입력 시 `Member` 생성
- 배송지 입력 시 `Member` 수정
- 설문 입력 시 `Survey` 생성 및 `Member`와 매핑
- 설문 상품 선택 시 `SurveyProduct` 매핑 엔티티 생성

이 구조의 문제는 **중간 단계부터 데이터가 확정된다는 점**이었다.
UI에서는 최종 완료 전까지 이전 단계 수정이 가능하지만, 백엔드에서는 이미 생성된 엔티티와 매핑이 존재했기 때문에 이전 단계로 돌아갈수록 데이터를 되돌리거나 다시 맞추는 비용이 커졌다.

단계 진입 자체는 AOP guard로 제어하려 했다.
각 단계 API에 커스텀 어노테이션을 달아 단계 순서를 강제하는 방식이었다.
그러나 이 접근에는 구조적인 한계가 있었다.

첫째로, **Spring AOP는 Proxy 기반이라 같은 클래스 내부에서 직접 호출하면 어노테이션이 적용되지 않는다(self-invocation 문제)**.
단계 검증 로직이 같은 서비스 클래스 안에 있으면 Proxy를 경유하지 않아 guard가 무시됐다.
Proxy를 거치도록 클래스를 추가로 분리하면 책임이 흩어지고 구조가 복잡해졌다.

둘째로, 설령 AOP가 정상적으로 동작하더라도 guard는 '다음 단계로 갈 수 있는지'를 막는 장치에 머물렀다.
이미 생성된 데이터를 임시 상태로 유지하거나, 최종 제출 전까지 확정을 미루는 문제는 guard만으로 해결할 수 없었다.

또한 이 상태를 프론트엔드 캐시만으로 관리하면 화면 이동 중에는 복원이 가능하더라도, 앱을 종료했다가 다시 진입했을 때 회원가입 진행 상태를 안정적으로 이어가기 어렵다.
회원가입을 최종 완료하기 전까지는 수정 가능한 임시 상태로 유지하면서도, 재진입 시 동일한 상태를 복원할 수 있어야 했다.

기존 모델은 **선형 상태 머신**을 기준으로 설계되어 있었고, **되돌릴 수 있고 이어서 작성할 수 있는 회원가입 UI 플로우**와는 정합성이 맞지 않았다.


---
## 💡 해결 방향

회원가입은 **Draft를 편집하다가 마지막에 Commit하는 트랜잭션**으로 재정의했다.

핵심은 다음 두 가지다.

1. 회원가입 진행 중에는 실제 도메인 엔티티를 확정하지 않는다.
2. 최종 완료 시점에만 필요한 엔티티를 한 번에 생성한다.

이를 위해 회원가입 전용 Aggregate Root인 `SignupDraft`를 두고, 기본 정보, 배송지, 설문 입력, 현재 단계 상태를 모두 Draft 내부에서 관리하도록 변경했다.


---
## 🔧 구현

`SignupDraft`는 회원가입 진행 중 필요한 상태를 한곳에서 관리하는 Aggregate Root다.

- `BasicInfo`
- `DeliveryInfo`
- `SurveyDraft`
- `currentStep`
- `completedSteps`

회원가입 진행 중에는 Draft 내부의 값만 수정한다.
사용자가 이전 단계로 돌아가더라도 이미 생성된 `Member`, `Survey`, `SurveyProduct`를 되돌리지 않고 Draft 값만 다시 편집하면 된다.

API도 Draft 중심으로 재구성했다.

- `POST /signup/draft`
- `PUT /signup/draft/basic`
- `PUT /signup/draft/delivery`
- `PUT /signup/draft/survey`
- `GET /signup/draft`
- `POST /signup/commit`

최종 `commit` 시점에는 Draft에 저장된 값을 바탕으로 실제 엔티티를 단일 트랜잭션으로 생성한다.
이 시점에 `Member`, `Survey`, `SurveyProduct`가 함께 확정되고, 회원가입 완료 전까지는 모든 입력이 임시 상태로 유지된다.

미완료 Draft가 계속 남지 않도록 만료 정책도 함께 두었다.

- `SignupDraftExpireJob`
- 스케줄러 기반 만료 Draft 정리


---
## ✅ 결과

회원가입 완료 전까지는 데이터를 임시 상태로 관리하고, 최종 완료 시점에만 확정하도록 구조를 바꾸면서 UI 플로우와 백엔드 모델의 정합성을 맞출 수 있었다.

이후에는 다음 효과를 얻었다.

- 이전 단계 수정 시 이미 생성된 엔티티를 되돌릴 필요가 없어짐
- 단계 진행과 데이터 확정 시점을 분리함
- 회원가입 완료 전 데이터의 성격을 명확히 Draft로 한정함
- 앱 재진입 시에도 진행 상태를 이어서 복원할 수 있게 됨

## 📝 정리

회원가입은 `SignupDraft` 기반의 Draft-Commit 구조로 재설계했고, 되돌릴 수 있는 UI 플로우와 데이터 확정 시점의 충돌을 해결했다.
