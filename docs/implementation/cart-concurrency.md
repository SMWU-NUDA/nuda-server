# 장바구니 동시성 제어

## 📋 배경

장바구니는 동일 상품을 여러 번 담을 수 있고, 이미 담긴 상품이라면 새 row를 만드는 대신 수량을 증가시키는 동작이 필요했다.
이 기능은 단순 저장처럼 보이지만, 같은 사용자가 짧은 시간 안에 연속으로 요청을 보낼 수 있기 때문에 동시성 제어가 필요했다.


---
## ⚠️ 문제 상황

초기 구현은 다음 흐름으로 동작했다.

- 회원의 장바구니 조회
- 장바구니가 없으면 새로 생성
- 상품이 있으면 수량 증가, 없으면 새 항목 추가

이 구조에서는 사용자가 장바구니 담기 버튼을 빠르게 두 번 누르거나, 거의 동시에 같은 요청이 들어왔을 때 문제가 발생할 수 있었다.

정상적인 기대는 다음과 같다.

- 첫 번째 요청: 상품 추가
- 두 번째 요청: 같은 상품 수량 증가
- 최종 결과: 수량 2

하지만 실제로는 첫 번째 요청만 성공하고, 두 번째 요청은 유니크 제약 조건 위반으로 500 오류가 발생할 수 있었다.
즉, 문제의 본질은 단순 중복 insert가 아니라, **같은 장바구니를 동시에 수정하는 요청을 어떤 기준으로 직렬화할 것인가**였다.


---
## 💡 해결 방향

장바구니는 다음 특성을 가진다.

- 트랜잭션이 짧다
- 동일 사용자 요청이 짧은 시간에 집중된다
- 실패 후 재시도보다 잠깐의 대기가 사용자 경험에 더 자연스럽다

이 기준으로 보면 낙관 락보다 비관 락이 더 적합했다.

- **낙관 락**
  - 충돌 후 retry 로직 필요
  - 실패 흐름이 비즈니스 로직까지 침범함
- **UNIQUE + 예외 처리**
  - 제약 조건 위반을 정상 제어 흐름처럼 다뤄야 함
  - 예외 중심 흐름이 됨
- **비관 락**
  - `Cart` 단위 직렬화 가능
  - 같은 회원의 동시 수정 요청을 순차적으로 처리 가능

즉, 장바구니에서는 '충돌 후 복구'보다 **충돌 자체를 대기로 직렬화하는 방식**이 더 자연스러웠다.

또한 락 기준점은 `CartItem`이 아니라 `Cart`여야 했다.
장바구니의 Aggregate Root가 `Cart`이기 때문에, 동시성 제어도 `Cart` 단위에서 이루어지는 편이 맞았다.


---
## 🔧 구현

### 1. `Cart` 조회 시 비관 락 적용

장바구니 조회 시점에 `PESSIMISTIC_WRITE`를 적용했다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select c from Cart c where c.memberId = :memberId")
Optional<Cart> findByMemberIdForUpdate(Long memberId);
```

이렇게 하면 **이미 존재하는 장바구니**에 대해서는 같은 회원의 동시 수정 요청이 직렬화된다.

### 2. 왜 `CartItem`에 락을 걸지 않았는가

`CartItem`에 락을 거는 방식은 적절하지 않았다.

- 아직 해당 상품 row가 없을 수 있다
- 중복 생성 타이밍을 막지 못한다
- 장바구니 전체 수정의 기준점으로 삼기 어렵다

반면 `Cart`는 장바구니의 Aggregate Root이므로, 같은 회원의 장바구니 변경 요청을 직렬화하는 기준점으로 삼기에 적절했다.

### 3. 비관 락만으로 해결되지 않았던 이유

비관 락은 **이미 존재하는 행에만 동작한다.**
즉, 장바구니가 없는 상태에서는 조회 결과가 `Empty`이므로 락 자체가 걸리지 않는다.

이 상태에서 여러 스레드가 동시에 들어오면 다음 흐름이 발생할 수 있다.

- 모든 스레드가 `Cart 없음`을 확인
- 동시에 `new Cart(memberId)` 생성 시도
- 유니크 제약 조건으로 하나만 성공, 나머지는 실패

즉, `find → empty → save` 경로에서는 비관 락만으로 생성 경쟁을 막을 수 없었다.

문제는 락 전략 자체보다, **장바구니 생성과 수정 책임이 같은 메서드에 섞여 있었다는 점**이었다.

### 4. 생성 책임 분리

최종적으로는 장바구니를 상품 추가 시점에 만들지 않고, 회원가입 시점에 미리 생성하도록 변경했다.

이 판단의 핵심은 단순했다.

- 상품 추가는 '수정'만 담당해야 한다.
- 장바구니 생성은 별도 시점에서 끝내야 한다.

이렇게 분리하면 장바구니 추가 로직은 더 이상 `생성 + 수정`을 동시에 다루지 않고, **기존 장바구니 수정**만 수행하게 된다.
즉, 동시성 제어 범위 자체를 줄인 것이다.


---
## ✅ 결과

이 구조를 적용한 뒤 장바구니 추가 로직은 동일 사용자 요청이 짧은 시간에 몰리더라도 `Cart` 단위 직렬화 안에서 처리될 수 있게 됐다.

또한 장바구니 생성 책임을 회원가입 시점으로 분리하면서 다음 효과를 얻었다.

- 상품 추가 로직에서 생성 경쟁 제거
- 유니크 제약 조건 위반으로 인한 500 오류 제거
- 장바구니 수정 경로의 동시성 제어 단순화
- Aggregate Root 기준의 락 전략 정립

결과적으로 사용자는 버튼 중복 클릭 상황에서도 예외를 마주하기보다, 의도한 대로 수량이 반영되는 흐름에 더 가깝게 동작하게 됐다.

## 📝 정리

장바구니는 동일 사용자 요청이 짧은 시간에 집중되는 특성을 고려해 `Cart` 단위 비관 락으로 직렬화하고, 생성 책임을 분리해 생성 경쟁까지 제거했다.
