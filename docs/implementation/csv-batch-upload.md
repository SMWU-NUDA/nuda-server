# 관리자 CSV 대량 업로드 배치 설계

## 📋 배경

관리자 백오피스에서는 상품 데이터를 한 번에 등록할 수 있는 CSV 업로드 기능이 필요했다.
이 기능은 단순 파일 업로드가 아니라, 대량 데이터 처리 성능, 참조 데이터 검증, 도메인 규칙 적용, 관리자 오류 확인까지 함께 고려해야 하는 작업이었다.


---
## ⚠️ 문제 상황

CSV 업로드는 일반적인 단건 저장과 성격이 다르다.
처리량이 크고, 검증 범위가 넓으며, 어느 시점에 실패를 감지할지도 미리 정해야 한다.

특히 1만 건 수준의 상품 데이터를 업로드할 경우 다음 문제가 동시에 발생할 수 있었다.

- 한 번에 모든 데이터를 메모리에 올리면 메모리 사용량이 커진다.
- 영속성 컨텍스트를 정리하지 않으면 flush 시점이 늦어지고 메모리 부담이 커진다.
- `Brand`, `Category`를 행마다 조회하면 N+1 쿼리로 인해 INSERT batching의 이점이 크게 줄어든다.
- 참조 데이터 오류를 늦게 발견하면 이미 앞선 데이터가 persist된 뒤에 실패를 마주하게 된다.
- 도메인 모델이 CSV 형식을 직접 알게 되면 입력 포맷이 도메인을 오염시킨다.

또한 이 기능은 저장 방식 자체에 대한 판단도 필요했다.

- 하나라도 실패하면 전체를 롤백할 것인가
- 일부만 성공시키고 실패 행을 분리할 것인가

이 구현에서는 관리자 백오피스의 데이터 정합성을 우선해 **전체 롤백 기준**으로 설계했다.
즉, 업로드 도중 유효하지 않은 행이 하나라도 발견되면 전체 요청을 실패로 간주하고 수정 후 다시 업로드하도록 했다.

문제의 본질은 CSV를 읽는 것이 아니라, **대량 입력을 도메인 규칙과 성능 제약 안에서 일관되게 처리하는 구조를 만드는 것**이었다.


---
## 💡 해결 방향

핵심 판단은 다음과 같았다.

1. **배치 처리의 핵심은 INSERT를 묶는 것만이 아니라, 조회를 배치 구간 밖으로 밀어내는 것이다.**
2. **참조 데이터 검증은 업로드 전에 끝내고, 배치 구간에서는 INSERT에만 집중해야 한다.**
3. **CSV 해석 책임과 도메인 생성 책임을 분리해야 한다.**
4. **대량 처리 환경에서는 영속성 컨텍스트를 명시적으로 관리해야 한다.**
5. **관리자 기능인 만큼 어떤 행에서 왜 실패했는지 즉시 알 수 있어야 한다.**

이를 바탕으로 CSV 파싱, 형식 검증, 참조 데이터 preload, 도메인 생성, batch insert를 단계별로 분리한 파이프라인으로 설계했다.


---
## 🔧 구현

전체 흐름은 다음과 같다.

- `ProductCsvReader`
  - `MultipartFile`을 읽어 `ProductCsvRow` 목록으로 변환
- `ProductCsvValidator`
  - 필수값, 형식, 값 범위 검증
- `ProductAdminService`
  - `Brand`, `Category` preload
  - 도메인 객체 생성
  - batch insert 수행

### 1. 입력 DTO와 도메인 책임 분리

CSV 한 줄은 `ProductCsvRow` record로 표현했다.
record를 사용한 이유는 입력 데이터를 불변 값으로 다루고, '이 객체는 비즈니스 로직이 아니라 입력 한 줄을 표현하는 값 묶음'이라는 역할을 분명히 하기 위해서다.

도메인 객체 생성은 CSV 자체를 직접 받지 않고, 서비스 계층이 해석한 순수한 값만 넘기도록 했다.

- 잘못된 방향
  - `Product.create(row)`
- 적용한 방향
  - `Product.create(brand, category, name, costPrice, discountRate, content, thumbnailImg)`

이렇게 분리한 이유는 도메인이 CSV 포맷을 직접 알지 않도록 하기 위해서다.
입력 포맷이 바뀌더라도 도메인 규칙은 그대로 유지할 수 있다.

### 2. 참조 데이터 preload

`Brand`, `Category`는 행마다 조회하지 않고 업로드 전에 한 번에 읽어 `Map`으로 올렸다.

- `Map<String, Brand>`
- `Map<CategoryCode, Category>`

이 판단은 단순한 N+1 방지 차원을 넘는다.

1. **N+1 쿼리를 막는다.**  
   1만 행 CSV에서 브랜드와 카테고리를 행마다 조회하면 조회 쿼리만 수만 번 발생한다.

2. **배치 구간의 순수성을 지킨다.**  
   INSERT를 batch로 묶더라도 중간에 SELECT가 row 단위로 섞이면 DB 왕복이 늘어나고, batch 처리의 의미가 약해진다.

3. **실패를 더 빨리 발견한다.**  
   참조 데이터가 잘못된 경우, 배치 도중 뒤늦게 실패하는 대신 업로드 초기에 곧바로 검증할 수 있다.

즉, preload는 조회 비용을 줄이는 최적화이면서, 잘못된 참조 데이터를 배치 초반에 걸러내기 위한 선택이기도 했다.

### 3. 배치 INSERT를 위한 JPA 설정

JPA batch insert가 실제로 동작하려면 ID 생성 전략부터 다시 봐야 했다.
`IDENTITY` 전략은 insert 시점에 키를 확정해야 하므로 batching에 불리하다.
따라서 상품 엔티티는 `SEQUENCE` 기반으로 전환하고, `allocationSize`를 조정해 batch insert가 가능하도록 바꿨다.

또한 다음 설정을 적용했다.

- `hibernate.jdbc.batch_size`
- `order_inserts`
- `order_updates`

저장 시에는 `EntityManager.persist()` 이후 `batch_size` 단위로 `flush()`와 `clear()`를 호출했다.
이렇게 하면 영속성 컨텍스트에 엔티티가 과도하게 쌓이는 것을 막으면서 insert를 묶어서 처리할 수 있다.

즉, **ID 생성 방식과 영속성 컨텍스트 정리 방식까지 함께 맞춰 실제로 batch insert가 동작하는 조건을 만든 것**이다.

### 4. 오류 설계

CSV 관련 오류는 전용 `CsvErrorCode`와 `CsvValidationException`으로 분리했다.
특히 `rowNumber`를 포함해 어느 행에서 어떤 문제가 발생했는지 바로 알 수 있도록 구성했다.

예:
- `CSV_INVALID_FORMAT`
- `CSV_MISSING_REQUIRED_FIELD`
- `CSV_INVALID_VALUE`
- `CSV_INVALID_REFERENCE`

이 구조를 통해 관리자 입장에서는 **어느 줄이 왜 실패했는지**를 바로 확인할 수 있다.

## 📊 측정 및 검증

성분 CSV 40,028행 기준으로 before/after를 측정했다.

| 항목 | 수치 |
|------|------|
| 테스트 데이터 | 40,028행 성분 CSV |
| 배치 처리 전 | 207.8s |
| 배치 처리 후 | 5.4s |
| 개선율 | 38.7배 단축 (97.4% 감소) |

배치 처리 전 구조는 `EntityManager.persist()` 후 flush를 행 단위로 수행하는 방식이었다.
INSERT 쿼리가 건 단위로 DB에 전달되고, 영속성 컨텍스트에 엔티티가 계속 누적되면서 flush 비용이 선형으로 증가했다.

배치 처리 후에는 `batch_size=500` 단위로 INSERT가 묶여 한 번의 JDBC 호출로 전달되고, `flush/clear`로 컨텍스트 비대화가 제거됐다.
PostgreSQL 드라이버의 `reWriteBatchedInserts=true`를 함께 설정해 `INSERT INTO ... VALUES (...), (...), (...)` 형태로 실제 배치 쿼리가 발행되도록 했다.

## ✅ 결과

이 구조를 적용한 뒤 CSV 업로드는 파일 파싱, 검증, 참조 데이터 조회, 도메인 생성, batch insert의 책임이 분리된 형태로 정리됐다.

구체적으로는 다음 효과를 얻었다.

- 참조 데이터 조회와 INSERT 구간을 분리해 배치 처리 성능을 확보함
- preload 전략으로 N+1 쿼리와 늦은 참조 오류 발견 문제를 함께 줄임
- `flush/clear`를 통해 대량 처리 시 영속성 컨텍스트 부담을 제어함
- 도메인이 CSV 포맷을 직접 알지 않도록 책임을 분리함
- 행 번호 기반 오류 메시지로 관리자 UX를 개선함

## 📝 정리

관리자 CSV 업로드를 단순 파일 저장 기능이 아니라, 대량 입력 처리 성능, 참조 데이터 검증, 도메인 순수성, 관리자 오류 확인까지 함께 고려한 배치 파이프라인으로 설계했다.
