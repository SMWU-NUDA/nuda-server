# ML 리뷰 분석 API 성능 최적화

## 📋 배경

상품 상세 화면에서는 리뷰 기반 요약 정보를 함께 제공하고 있었고, 이를 위해 ML 서버의 여러 분석 API를 조합해 사용하고 있었다.
이 기능은 탐색 경험을 높이는 데 유용했지만, 외부 ML 호출이 누적되면 응답 시간이 쉽게 길어질 수 있는 구조였다.


---
## ⚠️ 문제 상황

초기 리뷰 분석 API는 다음 세 가지 ML 엔드포인트를 순차적으로 호출하고 있었다.

- 리뷰 트렌드 분석
- 감성 분포 분석
- 키워드 추출 분석

각 API는 평균 40~60ms 수준의 응답 시간을 보였고, 이 구조에서는 세 호출의 대기 시간이 응답 경로에 그대로 누적될 수 있었다.
실측 결과 기준으로 캐시 미적용 시 전체 응답 시간은 JVM 내부 측정 약 72ms, 네트워크 포함 기준 약 80~90ms 수준이었다.

문제의 핵심은 두 가지였다.

1. 세 분석 작업은 서로 독립적인데도 직렬 호출로 묶여 있었다.
2. 동일 상품에 대한 분석 결과가 반복 호출될 때마다 외부 ML I/O 비용을 다시 지불하고 있었다.

결국 구조적으로는 **직렬 외부 호출**, 운영적으로는 **반복 계산**이 응답 시간을 지연시켰다.


---
## 💡 해결 방향

핵심 판단은 다음과 같았다.

1. **서로 독립적인 외부 호출은 직렬이 아니라 병렬로 바꿔야 한다.**  
   세 API가 서로의 결과를 기다릴 필요가 없기 때문에, 전체 응답 시간은 각 호출 시간의 합보다 최장 호출 시간에 가깝게 줄어들 수 있다.

2. **동일 상품에 대한 반복 분석은 다시 계산하지 않는 편이 낫다.**  
   리뷰 분석 결과는 상품 단위로 재사용 가능하므로, 반복 조회에 대해서는 캐시가 더 적합하다.

3. **성능 개선은 체감이 아니라 실측으로 검증해야 한다.**  
   내부 로그와 실제 네트워크 응답 시간을 함께 확인해 구조 개선 효과를 검증한다.

따라서 최적화의 1차 목표는 **직렬 외부 호출 구조를 병렬 구조로 바꾸는 것**, 2차 목표는 **반복 계산을 캐시로 제거하는 것**이었다.


---
## 🔧 구현

### 1. ML API 병렬 처리

트렌드, 감성, 키워드 분석은 서로 의존하지 않는 작업이므로 `CompletableFuture`를 사용해 병렬 호출하도록 구성했다.

이렇게 바꾸면 세 작업을 순차적으로 기다리지 않고, 전체 응답 시간은 가장 오래 걸리는 호출 시간을 중심으로 수렴한다.
병렬화는 응답 시간의 상한을 바꾸는 구조적 변경이었다.

### 2. Redis 캐시 도입

병렬화 이후에도 동일 상품에 대한 반복 조회는 외부 ML 호출 비용을 계속 발생시킨다.
이를 줄이기 위해 분석 결과를 Redis에 캐시했다.

- Key 형식: `ml:review:{type}:{productId}`
- TTL 기반 만료 정책 적용
- `GenericJackson2JsonRedisSerializer` 사용

#### record DTO 역직렬화 이슈

ML 응답을 담는 DTO를 Java `record`로 선언했는데, Redis에서 꺼낼 때 역직렬화 오류가 발생했다.

원인은 `GenericJackson2JsonRedisSerializer`가 내부적으로 Jackson을 사용하는데, Jackson은 기본적으로 역직렬화 시 no-arg 생성자를 찾기 때문이다.
Java `record`는 no-arg 생성자가 없고, 컴포넌트 파라미터를 받는 canonical 생성자만 존재한다.

이를 해결하기 위해 DTO에 `@JsonCreator`를 명시해 Jackson이 canonical 생성자를 통해 역직렬화하도록 지정했다.

```java
public record ReviewTrendResponse(
        @JsonProperty("keywords") List<String> keywords,
        @JsonProperty("trend") String trend
) {
    @JsonCreator
    public ReviewTrendResponse { }  // canonical 생성자를 @JsonCreator로 위임
}
```

이 수정 이후 캐시 저장 → 역직렬화 → 재사용이 안정적으로 동작했다.

### 3. 캐시 HIT 시 동작

캐시 HIT 시에는 다음 비용이 제거된다.

- ML 서버 호출
- 외부 네트워크 I/O
- 추가적인 연산 처리

동일 상품에 대한 반복 조회에서는 캐시가 응답 시간을 더 직접적으로 줄여준다.
캐시는 병렬화된 외부 호출 자체를 반복 경로에서 제거하는 두 번째 계층으로 동작한다.

- 네트워크 포함 기준 약 6배 개선

## ✅ 결과

구조 적용 후 실제 응답 시간은 다음과 같이 개선됐다.

- 캐시 미적용: 약 80~90ms
- 캐시 적용: 평균 14.5ms

네트워크 포함 기준으로 약 6배 이상의 응답 시간 단축을 확인했다.

이 개선의 핵심은 다음 네 가지였다.

- 독립적인 ML 분석 작업을 병렬 호출 구조로 전환
- 반복 조회 경로에서 외부 ML 호출 제거
- Redis 직렬화/역직렬화 안정화
- 내부 로그와 실제 `curl` 측정을 함께 사용한 검증

이 작업은 **직렬 외부 호출 구조를 병렬 구조로 재설계하고, 반복 계산을 캐시로 제거한 성능 개선**으로 정리할 수 있다.

## 📝 정리

ML 리뷰 분석 API는 세 개의 독립적인 외부 분석 작업을 병렬화해 응답 구조를 먼저 개선하고, Redis 캐시를 통해 반복 조회 비용까지 제거하는 방식으로 최적화했다.
