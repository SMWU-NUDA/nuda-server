# 이벤트 후처리와 ML 추론 실행 풀 분리

## 📋 배경

서비스에는 회원 탈퇴 후처리, 키워드 변경 이벤트, 리뷰 요약처럼 트랜잭션 이후에 이어지는 작업이 함께 존재한다.
이 작업들은 모두 요청 이후에 실행되지만, 응답 지연에 미치는 영향과 실행 비용은 서로 달랐다.


---
## ⚠️ 문제 상황

초기에는 후처리 작업을 요청 흐름과 명확히 분리하지 않으면, HTTP 요청을 처리하는 Tomcat 스레드가 트랜잭션 이후 작업까지 오래 점유할 수 있었다.
이 경우 사용자는 이미 응답을 받을 수 있는 상황에서도, 후처리 때문에 요청 스레드가 반환되지 않아 전체 응답성이 나빠질 수 있다.

문제는 후처리 작업을 전부 같은 방식으로 처리할 수 없었다는 점이다.
실제로 후처리 안에서도 작업마다 실행 비용과 처리 시점이 달랐다.

- 탈퇴 요청 직후의 일부 정리 작업은 요청 흐름 안에서 바로 끝내는 편이 자연스러웠다.
- 키워드 변경, 커머스 이벤트 같은 작업은 응답 이후 비동기로 넘겨도 무방했다.
- 리뷰 요약처럼 ML 서버 호출이 필요한 작업은 일반 이벤트 후처리보다 훨씬 무겁고, 외부 의존성까지 포함하고 있었다.

이 세 종류를 같은 방식으로 처리하면 두 가지 문제가 생긴다.

1. 모든 작업을 동기로 처리하면 요청 스레드가 오래 점유된다.
2. 모든 작업을 같은 비동기 풀에 넣으면, 무거운 ML 작업이 큐를 점유하면서 가벼운 이벤트 후처리까지 함께 지연된다.

핵심은 **성격이 다른 작업을 같은 실행 경로에 묶어두면 병목이 전파된다는 점**이었다.


---
## 💡 해결 방향

실행 경로를 작업 성격에 따라 분리했다.

핵심 판단은 다음과 같다.

1. **사용자 응답 경로는 가장 짧아야 한다.**  
   요청 처리와 트랜잭션은 Tomcat 스레드에서 끝내되, 응답 이후에도 가능한 작업은 요청 경로에서 분리한다.

2. **작업 성격에 맞는 처리 시점을 구분한다.**  
   요청 직후 바로 마무리되어야 하는 정리 작업은 동기 처리하고, 응답 이후 처리 가능한 작업은 비동기로 넘긴다.

3. **일반 이벤트 후처리와 ML 추론은 별도 실행 경로로 관리한다.**  
   ML 호출은 외부 네트워크 I/O와 연산 비용이 커서, 일반 이벤트 큐와 분리하지 않으면 병목을 전파시킬 가능성이 높다.

이 구조의 목적은 **HTTP 응답 경로를 보호하고 무거운 작업의 병목이 다른 처리로 번지지 않게 만드는 것**이었다.


---
## 🔧 구현

전체 실행 경로는 세 계층으로 나눴다.

- **Tomcat 스레드 풀**
  - HTTP 요청 처리
  - 컨트롤러, 서비스, 트랜잭션 수행
- **eventExecutor**
  - 이벤트 후처리 전용 실행 풀
  - `core=5`, `max=10`, `queue=50`
- **mlExecutor**
  - ML 추론 전용 실행 풀
  - `core=10`, `max=20`, `queue=100`

실제 흐름은 다음과 같이 분리했다.

### 1. 일반 API 요청
회원 탈퇴 요청처럼 일부 작업은 요청 직후 바로 정리되어야 했다.
이 경우 `WithdrawRequestEventHandler`는 `@Async` 없이 동기적으로 실행하고, `REQUIRES_NEW` 트랜잭션 안에서 처리했다.

이 판단의 이유는 분명했다.
이 작업은 ML 추론처럼 무겁지 않고, 요청 직후 바로 일관성을 맞추는 편이 자연스러운 작업이기 때문이다.
이 작업은 요청 흐름 안에서 짧게 마무리하는 편이 더 적절했다.

### 2. 이벤트 후처리
최종 탈퇴 확정 이후 후처리, 커머스 이벤트, 키워드 변경 이벤트는 `eventExecutor`로 위임했다.

이 작업들은 요청 응답보다 우선순위가 낮고, 사용자 응답을 붙잡고 있을 이유가 없었다.
따라서 요청 스레드에서 분리해 응답성을 보호하는 쪽이 더 중요했다.

### 3. ML 추론
키워드 변경 이후 리뷰 요약이나 추천 관련 후속 작업은 다시 `mlExecutor`로 분리했다.

이렇게 한 이유는 일반 이벤트 후처리와 ML 추론의 비용 차이가 컸기 때문이다.
ML 호출은 외부 서버와의 통신, 네트워크 대기, 상대적으로 큰 연산 시간을 포함하므로, 같은 풀에 넣으면 eventExecutor의 큐를 잠식할 수 있다.
따라서 ML 작업은 별도 전용 풀에서 처리하도록 격리했다.


---
## ✅ 결과

이 구조를 적용한 뒤 요청 스레드는 HTTP 요청과 트랜잭션 처리에 집중하고, 응답 이후 작업은 목적별 실행 풀에서 처리하도록 분리할 수 있었다.

구체적으로는 다음 효과를 얻었다.

- 요청 응답 경로가 무거운 후처리에 의해 지연되지 않음
- 일반 이벤트 후처리와 ML 작업이 서로 같은 큐를 점유하지 않음
- 외부 ML 호출이 후처리 전체의 병목으로 번지는 것을 완화함
- 작업 성격에 따라 동기/비동기 기준을 명확히 구분함

또한 `eventExecutor`와 `mlExecutor`의 큐 한계가 분리되면서, 특정 종류의 작업이 몰릴 때 어느 계층에서 병목이 발생하는지 더 명확하게 관찰할 수 있게 됐다.

## 📝 정리

이벤트 후처리와 ML 추론은 요청 처리, 일반 후처리, 무거운 외부 연산으로 나눠 각각 다른 실행 경로에서 처리했고, 그 구조를 통해 응답성과 안정성을 함께 확보했다.
