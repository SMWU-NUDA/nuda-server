# Elasticsearch 동기화 실패 격리

## 📋 배경

서비스는 상품 검색 기능에 Elasticsearch를 사용하고 있으며, 애플리케이션 기동 시 DB에는 존재하지만 Elasticsearch에는 없는 상품을 찾아 초기 인덱싱을 수행하고 있었다.
이 작업은 검색 품질을 유지하는 데 필요했지만, 사용자 요청을 받기 위한 애플리케이션 기동 자체보다 우선순위가 높은 작업은 아니었다.


---
## ⚠️ 문제 상황

초기 구현에서는 이 동기화 로직을 `ApplicationRunner`에서 실행하고 있었다.
문제는 이 구조가 **검색 계층의 실패를 애플리케이션 전체 기동 실패로 확장시킬 수 있었다**는 점이다.

Elasticsearch는 검색 기능을 담당하는 보조 계층이며, 일시적인 장애가 발생할 수 있다.
초기 인덱싱은 중요하지만, 인증이나 주문처럼 서비스 전체 기동을 멈출 정도의 핵심 경로는 아니었다.
그럼에도 `ApplicationRunner` 안에서 예외가 그대로 전파되면, ES 일시 장애가 전체 서비스 가용성 저하로 번질 수 있었다.


---
## 💡 해결 방향

핵심 판단은 다음과 같았다.

1. **검색 인덱스 동기화는 중요하지만, 애플리케이션 기동을 막아서는 안 된다.**
2. **기동 경로와 비핵심 초기화 작업을 분리해야 한다.**
3. **일시적인 ES 장애는 자동 재시도로 흡수하고, 최종 실패는 격리된 형태로 남겨야 한다.**

이를 위해 동기화 시점을 `ApplicationRunner`에서 제거하고, 애플리케이션이 완전히 올라온 뒤 비동기적으로 실행하도록 구조를 바꿨다.
또한 재시도 가능한 실패는 자동 재시도하고, 재시도 소진 후에는 애플리케이션을 중단시키지 않고 오류를 로그로 남기도록 했다.


---
## 🔧 구현

구조는 두 단계로 분리했다.

### 1. 기동 경로 분리
동기화 시작 시점을 `ApplicationReadyEvent` 이후로 옮기고, `@Async("eventExecutor")`를 적용해 메인 기동 스레드와 분리했다.

이렇게 하면 애플리케이션은 먼저 정상 기동을 완료하고, 검색 인덱스 동기화는 별도 실행 경로에서 수행된다. 이로써 Elasticsearch가 일시적으로 불안정하더라도 서비스 자체는 요청을 받을 수 있다.

### 2. 재시도와 실패 격리
실제 동기화 로직은 별도 클래스로 분리하고 `@Retry(name = "es-sync")`를 적용했다.

- `ProductSearchSyncRunner`
  - `ApplicationReadyEvent` 수신
  - 비동기 실행 시작 책임
- `ProductSearchSyncExecutor`
  - 실제 동기화 수행
  - `@Retry` 적용
  - fallback 처리

이렇게 분리한 이유는 Spring AOP의 self-invocation 문제 때문이다.
`@Retry`는 프록시 기반으로 적용되기 때문에, 같은 클래스 내부에서 직접 `sync()`를 호출하면 retry가 동작하지 않는다.
따라서 기동 트리거 역할과 실제 실행 역할을 분리해, 동기화 호출이 반드시 프록시를 거치도록 구성했다.

동기화 실패 시에는 재시도 정책을 적용한다.

- 최대 3회 재시도
- 지수 백오프(초기 1s, 배수 2배) 기반 대기
- 재시도 소진 시 fallback 실행

fallback은 애플리케이션을 중단시키지 않고, ERROR 레벨 로그를 남긴다.

```
[ES Sync] 초기 동기화 최종 실패 — 수동 인덱싱 필요
실패 상품 수: {count}, 원인: {exception message}
```

이 로그는 Prometheus/Grafana 알럿과 함께 확인할 수 있도록 남겨, 운영자가 수동 인덱싱이 필요한지 판단할 수 있게 했다.
이 구조에서는 일시적인 장애를 재시도로 먼저 흡수하고, 끝까지 복구되지 않은 실패만 운영자가 확인하고 대응할 수 있도록 별도로 남긴다.


---
## ✅ 결과

이 구조를 적용한 뒤 Elasticsearch 초기 동기화는 검색 계층의 보조 작업으로 격리되었고, ES 장애가 애플리케이션 전체 기동 실패로 번지지 않도록 분리할 수 있었다.

구체적으로는 다음 효과를 얻었다.

- 애플리케이션은 정상 기동을 완료한 뒤 요청을 받을 수 있음
- 검색 동기화는 백그라운드에서 별도로 수행됨
- 일시적인 ES 장애는 재시도로 흡수 가능함
- 재시도 소진 후에도 애플리케이션은 살아 있고, 운영 로그를 통해 후속 대응 가능함
- 기동 경로와 비핵심 초기화 작업의 책임이 분리됨

## 📝 정리

Elasticsearch 초기 동기화를 기동 필수 경로에서 분리하고, `ApplicationReadyEvent + Async + Retry + fallback` 구조로 감싸 검색 계층의 일시 장애가 전체 서비스 가용성으로 번지지 않도록 설계했다.
